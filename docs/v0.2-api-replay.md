# v0.2 架构改造：API Capture + Replay

> 把 webXport 从「录制 click 重放」升级为「录制 API + 参数化重放」。
>
> 这是产品**真正的 PMF 落点**——AI agent 能任意指定日期 / 范围抓取，平台 UI 改版不影响录制脚本，周末出门也能补数据。

## v0.2 之后的产品定位

webXport 的本质升级为：「**用户定义工具的 MCP server**」（user-defined-tools MCP）。

- 用户录一次 → 这次录制变成 AI 可调用的"工具"（带参数 schema）
- AI 通过 MCP 调用，自由传参（日期 / 范围 / 过滤条件）
- 录制时同时捕获 **API 调用** + **click 序列**，重放时优先调 API（快、参数化、抗 UI 改版），失败 fallback click

**这不是「AI 看页面」，是「AI 用用户给它的工具」**。和 Browser Use / Computer Use / Stagehand 这类通用浏览器 agent **架构不同、定位不同**。详细见 [CLAUDE.md 心智模型](../CLAUDE.md#心智模型不要再走偏) 和 [decisions.md](decisions.md)。

## 一句话差异

| | v0.1.x（现状） | v0.2（目标） |
|---|---|---|
| 脚本本质 | 点击序列（DOM selector 数组） | API 调用模板（URL + method + body + headers） + 参数 spec |
| 重放方式 | 在 tab 里模拟点击 → 等 UI 触发下载 | 在 tab 里直接 `fetch(api, {credentials:'include'})` 拿 blob |
| 日期可变吗 | ❌ 录死 | ✅ 自动识别日期字段标为变量，AI / 用户传参 |
| 多天补跑 | ❌ 同样的"昨天"按钮点 N 次 | ✅ 同一个 API 模板，N 个日期参数，N 次调用 |
| 抗 UI 改版 | ❌ 按钮位置变就废 | ✅ API 不变就能跑 |

## 架构对比

### v0.1.x 流程

```
[录制]
  Content Script (recorder.ts)
    监听 click/input/change → 抓 selector → 发 rec/step → Background

[重放]
  Background.runScript()
    chrome.scripting.executeScript → 注入 replayer
    replayer 按 selector 找元素 → click/input → 等 DOM 稳定
    chrome.downloads.onDeterminingFilename → 归档文件
```

### v0.2 流程

```
[录制]
  Content Script (recorder.ts)
    同上，仍然抓 click（作为兜底）

  Background.recording.ts (新)
    chrome.webRequest.onBeforeRequest / onBeforeSendHeaders / onResponseStarted
      → 监听本 tab 所有 outgoing 请求（URL/method/body/headers/响应类型）
    chrome.downloads.onCreated 触发时
      → 找最近一个响应类型为 binary/excel/json 的请求 → 认定为"导出 API"
    把 API call (URL+method+body+headers) 保存到 Script.apiCalls

  Script Detail Popup
    显示捕获到的 API 调用
    自动识别 URL/body 里的日期字段 (regex 检测 YYYY-MM-DD / YYYYMMDD / Unix ms)
    标为参数 {date}，用户可改 / 确认

[重放]
  Background.runScript(script, params?)
    if script.apiCalls && !forceClick:
      apiReplay(script, params)  ← 默认路径
    else:
      clickReplay(script)         ← 兜底（老脚本 / API 调用失败）

  apiReplay(script, params)
    把 {date} 等参数替换进 URL/body
    chrome.scripting.executeScript → 在 tab 里执行：
      fetch(url, {method, body, headers, credentials:'include'})
      → resp.blob()
      → btoa(...)
      → 传回 Background
    Background 把 base64 解码 → 用 chrome.downloads.download(data url, filename) 写盘
```

## 核心类型

```typescript
// src/shared/types.ts

export interface ApiCall {
  /** HTTP method: GET / POST */
  method: string;
  /** URL 模板，含 {param} 占位符 */
  urlTemplate: string;
  /** Body（如有），JSON 字符串模板，含 {param} 占位符 */
  bodyTemplate: string | null;
  /** 必要的 headers（如 Content-Type / X-Custom-Token） */
  headers: Record<string, string>;
  /** 期望的响应类型（excel / csv / json / binary），用于校验 */
  expectedContentType: string;
}

export interface ParamSpec {
  /** 参数名，如 "date" / "startDate" / "endDate" */
  name: string;
  /** 参数类型 */
  type: 'date' | 'date-range' | 'string';
  /** 默认值表达式：'yesterday' / 'today' / 'today-7' */
  defaultExpr: string;
  /** 日期格式（仅 date 类型）：'YYYY-MM-DD' / 'YYYYMMDD' / 'unix-ms' / 'unix-s' */
  format?: string;
}

export interface Script {
  // 原有字段...
  
  /** v0.2 新增：API 调用模板（可能多个，比如 千帆模式：submit → poll → download） */
  apiCalls?: ApiCall[];
  /** v0.2 新增：参数规约 */
  params?: ParamSpec[];
}
```

## 分阶段实施

每阶段独立可测试、可 commit。

### Phase 1：API 捕获（5-7 天）

**目标**：录制时识别出"导出 API"，把请求元数据存进脚本。**重放还是走 click 路径**——只是多存了 API 数据看着。

任务：
- [ ] `manifest.json` 加 `webRequest` permission
- [ ] `src/background/api-capture.ts`：start/stop 接口，挂 webRequest 监听
- [ ] 关联逻辑：chrome.downloads.onCreated 触发时，找最近一个 onBeforeRequest（同 tab + 时间窗 5 秒内 + 响应非 HTML）
- [ ] 保存到 Script.apiCalls
- [ ] popup 脚本详情页加一栏「捕获的 API」展示（read-only）

**验收**：录制一个小红书 / 抖店导出脚本，popup 详情能看到 `POST /api/export?date=2026-05-15` 这样的 API 信息。重放仍然走旧路径（click）。

### Phase 2：参数识别（2-3 天）

**目标**：自动识别 URL / body 里的日期字段，转成 `{date}` 占位符。

任务：
- [ ] `src/shared/param-detect.ts`：regex 扫描 `YYYY-MM-DD` / `YYYYMMDD` / Unix ms / Unix s（最近 N 天范围）
- [ ] 录制完成后自动转换：URL `?date=2026-05-15` → `?date={date}`，生成 ParamSpec
- [ ] popup 加「参数」面板：列出参数 + 默认值表达式（如 `yesterday`），用户可改
- [ ] 默认值求值：`yesterday` → today - 1 day，format 按 ParamSpec.format 输出

**验收**：录制时填的是"昨天"日期，自动识别为 `{date}`，默认 `yesterday`。手动改成 `today` 重放，URL 里日期就变成今天。

### Phase 3：API 重放（5-7 天）

**目标**：重放时优先调 API（不点击），失败再 fallback。

任务：
- [ ] `src/background/api-replay.ts`：
  - 输入 ApiCall 数组 + params
  - 替换占位符
  - chrome.scripting.executeScript 在目标 tab 注入：`await fetch(url, {credentials:'include', method, body, headers})` → blob → base64
  - 接收 base64 → chrome.downloads.download({ url: 'data:...;base64,...', filename })
  - 走原有 onDeterminingFilename 归档逻辑
- [ ] `runner.ts`：调用顺序 `apiReplay → 失败则 clickReplay`
- [ ] `RunResult` 加 `mode: 'api' | 'click' | 'mixed'`
- [ ] 错误分类：401（session 过期）/ 4xx（参数错）/ empty blob → fallback；网络错 → 直接报错

**验收**：录制好的脚本走 API 重放，速度比 click 快 5-10 倍，跑 3 次不同日期能拿到 3 份不同数据。

### Phase 4：多步 API 链（5-7 天）

**目标**：支持 submit-task → poll → fetch-CDN 这种 3 段式（小红书千帆 / 抖店罗盘）。

任务：
- [ ] api-capture 识别"任务型导出"模式（请求 → 轮询 → 文件 URL）
- [ ] 保存为 ApiCall 数组（多个）
- [ ] api-replay 按顺序执行；poll 步骤识别"是否完成"字段（自动从录制时观察）
- [ ] 把动态 URL（如 poll 返回的 CDN 地址）作为变量传递给下一步

**验收**：小红书千帆完整导出流程能跑通。

### Phase 5：MCP 参数化（1-2 天）

**目标**：AI agent 能传日期 / 范围参数。

任务：
- [ ] `mcp-server/src/tools.ts`：`run_script` schema 加 optional `params` object
- [ ] `list_scripts` 返回值带 `acceptedParams: ParamSpec[]`，让 AI 知道该传啥
- [ ] WS bridge RPC 透传 params 字段
- [ ] runner 接收 params 并应用
- [ ] Tool 描述更新："you can pass params={date:'2026-05-13'} to backfill historical data"

**验收**：在 Claude Code 里说"拉上周一到周五五天的小红书数据"，AI 自动调用 5 次 run_script 带不同 date，5 个文件都到位。

### Phase 6：迁移 + 兜底（2-3 天）

**目标**：老脚本平滑、API 失效时不丢用户。

任务：
- [ ] 老脚本（无 apiCalls）继续走 click 重放，0 迁移
- [ ] popup 里给老脚本一个"重新录制升级到 API 模式"按钮
- [ ] API 重放连续 3 次失败 → 自动标记为"降级到 click 重放"
- [ ] popup 上显示"⚠️ 此脚本运行在降级模式，建议重录"

## 风险 & 对策

### 风险 1：部分平台 API 带 CSRF / 动态签名（淘宝 / 京东系）

**症状**：直接 fetch 返回 401 或 403。

**对策**：
1. 录制时观察 headers，捕获 `X-Csrf-Token` / `X-Sign` 等自定义头
2. 部分签名是 JS 在请求前生成的（如 sign(payload + timestamp + secret)）—— 这种 fetch 重放会失败
3. **失败自动 fallback 到 click 重放**——保证产品体验不退步
4. 文档里诚实标注「淘宝系平台 API 模式可能 fallback」

### 风险 2：多步导出链路复杂（千帆 submit→poll→download）

**症状**：录制时抓到了第一段 POST 但漏了后续两段。

**对策**：
- Phase 4 专门做这个
- 启发式：onBeforeRequest 后 N 秒内同源同 cookie 的请求都收集 → 形成 chain
- 如果完全失败，仍然能 fallback click

### 风险 3：日期格式识别误判

**症状**：把非日期数字（订单号末 8 位刚好是 20260515）误识为日期。

**对策**：
- 只识别 URL **query 参数值** + **body JSON 值**，不识别路径
- 加白名单：字段名含 `date` / `time` / `start` / `end` / `day` 才考虑
- 用户可在 popup 编辑 ParamSpec，关闭误识

### 风险 4：响应类型识别错

**症状**：把请求 user info 的 GET 当成了导出 API。

**对策**：
- 看响应大小（导出文件通常 > 1KB）
- 看响应类型（excel / csv / zip / octet-stream 优先）
- 看响应是否触发 chrome.downloads.onCreated（最强信号）
- 实在不确定就把 top-3 候选交给用户选

### 风险 5：v0.2 改太大，v0.1 用户回不去

**对策**：
- 老脚本结构（无 apiCalls）保持兼容
- 重放路径自动判断：有 apiCalls 走 API，无则走 click
- 不强制迁移，用户用着舒服可以一直老脚本

## 与 Stage 0 launch 的关系

**v0.1.0 仍然按原计划 launch**：

- ✅ Chrome / Edge 审核继续走（已提交，等审就好）
- ✅ Stage 0 发售包给朋友圈 / V2EX / 即刻 / 微信群—— v0.1.0 已足够卖
- ✅ 收第一笔钱

**v0.2 开发并行**：

- 不阻塞 launch
- 用 v0.1 真实用户反馈验证 v0.2 设计假设（哪些平台真有 API，哪些必须 click）
- v0.2 完成后作为升级版给已付费用户免费推送

**时间预估**：6 阶段合计 **3-4 周**专心开发。如果 Stage 0 反馈频繁要响应，可能拉到 4-6 周。

## 商业模式 implications

v0.2 完成之后：

| | v0.1（click） | v0.2（API + click） |
|---|---|---|
| 价格 | ¥30 / 月 | **¥50-99 / 月**（差异化大幅提升） |
| 卖点 | "录一次每天自动跑" | "AI 能任意调度 + 历史回填 + 平台改版不挂" |
| 竞品对比 | 跟影刀 / Browse AI 同维度 | **单纯 AI agent 数据抓取层，无直接竞品** |
| 用户量级 | 数据运营 / 自媒体 | 加上独立开发者 / 数据分析师 / 海外用户 |

v0.2 才是 webXport 真正能值钱的版本。v0.1 是为了开始收钱 + 验证需求。

## 实施前需要确认的 open questions

1. **平台范围**：v0.2 先支持哪几个平台？建议优先级：
   - 小红书千帆（你日常在用，多步链路最复杂，做通了其他都简单）
   - 抖店罗盘（已有数据）
   - 淘宝（有 CSRF，验证 fallback 机制）
   - 视频号助手（简单 API，验证基础路径）

2. **录制 UI 变化**：Phase 1 之后录制 popup 要不要加一个"正在监听 API"的状态指示？
   建议：加（提高用户对内部行为的可见性）

3. **Phase 1 验收方式**：写小红书千帆脚本测试，还是先做个简单平台（如某博客后台导出）做 PoC？
   建议：先做一个简单平台 PoC，验证 webRequest + downloads 关联逻辑 OK，再上千帆

4. **fallback 触发条件**：fetch 返回 401 自动 fallback，还是给用户选择？
   建议：自动 fallback + 日志记录 + popup 提示，不打扰用户

5. **现有的 v0.1.x 在跑了——v0.2 开发分支吗？**
   建议：开 `v0.2` 分支，v0.1.x bug fix 仍在 main，v0.2 稳定后 merge
