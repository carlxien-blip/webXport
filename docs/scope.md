# 当前范围（V1）

webXport V1 = **Chrome 扩展 + MCP server + 后端**，三件套合起来构成"用户登录态浏览器作为 AI agent 数据抓取工具"的最小闭环。

定位：**AI agent（Claude Code / Cursor / Codex 等）通过 MCP 调用本扩展去抓取登录态后台数据**。AI agent 不在产品里，它是产品的消费方。

V1 分两阶段执行，**先把单用户场景跑稳，再上多租户**：

- **P1（单用户跑通）** — 作者自己的 Claude Code 在任意 VS Code window 都能稳定调用扩展抓数据；扩展 UX 补全到能日常用的程度。**不上后端、不做 license、所有功能默认开放**。
- **P2（多租户 + 付费）** — 加后端、账号、license 校验、付费墙。P1 没问题再开 P2，否则带着不稳定的链路上多租户只会乱套。

---

## P1 — 单用户跑通

### 扩展（Chrome MV3）

1. **录制**：popup → 选 tab → 操作 → 停止 → 命名保存（已有）
2. **重放**：手动触发 / `chrome.alarms` 定时触发 / MCP 远程触发（已有）
3. **下载归档**：监听 `chrome.downloads`，按业务时间归类到 `Downloads/webxport/{脚本名}/{日期}/`（已有）
4. **运行历史**：最近 10 次成功 / 失败 + 原因，存 `chrome.storage.local`（已有）
5. **脚本本地导入 / 导出 JSON**（待做）：备份 + 跨设备迁移基础款

> P1 不会再加任何"看着很有用"的功能。具体砍掉了什么、为什么砍 → 见下面「永远不做」。

### MCP server（已有，需固化）

- stdio ↔ WebSocket 桥，让 Claude Code / Cursor / Codex 等 MCP-capable agent 能调用扩展
- 4 个工具：`list_scripts` / `run_script(name, waitForResult)` / `get_runs(name?)` / `get_run_status`
- P1 阶段：**不做 license 校验**，跑通即用

### 安装脚本（新）

- 一条命令搞定：`./install.sh`
- 自动 build 扩展 + build mcp-server + 把 mcp 注册到 user-scope（`claude mcp add -s user`）+ 起 Chrome 加载扩展提示
- 替代之前那一长串"先 npm install 再 build 再 mcp add 再 reload"的设置仪式
- 这一步不为别人做，先为自己做——多租户化时它会演化成正式 installer

### P1 完成定义

- 作者在任意 VS Code window 跑 Claude Code，问"列出我的 webxport 脚本"能稳定返回
- 脚本能从 popup 导出 JSON，在另一台电脑导入恢复
- 重装 / 换电脑 ≤ 5 分钟搞定（跑一遍 `./install.sh` + 加载扩展）
- 完整功能能对应 [docs/guide.md](guide.md) 描述的体验

## P2 — 多租户 + 付费

### 后端（最简形态）

- Cloudflare Workers + D1（零运维、免费层够用）
- GitHub OAuth 登录（不自建邮件服务）
- 两张表：`users` + `license_keys`
- `/api/verify`：扩展用 license_key 换 token（编 tier + 过期时间），扩展缓存 24h
- 一个 admin 页（密码保护）：用户列表 + 手动发码按钮
- 扩展启动时调 verify → 失败 / 无 license = 默认免费 tier

### 扩展端

9. **License 状态展示 + 输入框**：试用期倒计时、付费状态、激活码粘贴框
10. **MCP 拒绝调用**：未授权时 MCP 接口返回明确升级提示

### 跨设备脚本云同步（付费版）

- 扩展登录态 → 脚本写本地 + 写云端
- 多设备登同一账号自动拉取脚本

### P2 完成定义

- 新用户能通过 GitHub OAuth 注册并默认进入 14 天 MCP 试用
- 试用 / 付费状态在扩展里清晰可见
- 后端不可达扩展也能用（7 天宽限期）
- 作者能给付费用户手动发激活码并在扩展里激活
- 付费用户多设备同步脚本

## 永远不做（架构决策，不要再问）

> 这些不是"V1 还没做"或"V2 也许做"——是**架构和产品定位决定了它们永远不在 webXport 里**。LLM 写代码、产品讨论时都按这个边界走，谁问都一样的回答。

| 永远不做 | 为什么 |
|---|---|
| **扩展内调用任何 LLM**（包括"selector 全失败时兜底"） | AI 永远是 *调用方* ——agent 通过 MCP 用我们，我们不主动调它。隐私、成本、不确定性、不可逆动作风险，全部不接受 |
| **官方脚本库 / 内置模板** | 平台 UI 一改模板就废；每个用户要的报表都不同；维护成本爆炸 |
| **步骤编辑器**（删 / 改 / 调序 / 插入 wait） | 典型脚本 5-10 步，重录 1-2 分钟。改 selector 违反"不模糊猜测"红线；插入 sleep 违反"自动 wait"红线。错了重录就行 |
| **失败截图** | run 历史已经记录"第 N 步失败：错误信息"，足够用户判断"重录 / 重新登录 / 关掉弹窗"。截图带不来不同的行动 |
| **Webhook / 飞书 / 钉钉 / 邮件 / 桌面通知** | 不下场做下游集成。要推送让用户自己用 AI agent 接（"跑完发我飞书"是 agent 的事不是插件的事） |
| **错过窗口补跑** | 半夜电脑关机 → 第二天默默跑了 = 诡异行为。Chrome 没开就不跑，用户开机自己手动跑 |
| **参数化录制 / 变量替换** | 用户用"昨天 / 今天"录死即可。要变量化让 AI agent 在调用 `run_script` 时改 |
| **脚本版本管理** | 重录覆盖即可，平台改一次旧版就废了，留着没意义 |
| **共享脚本网络 / 脚本市场** | 每个用户录自己的；网络效应不是本产品命题 |
| **数据分析 / 可视化 / 报表生成** | AI agent 的事 |
| **自动登录 / 验证码处理 / cookie 注入** | 前提就是用户已登录；做这个等于把命根子（登录态护城河）拆了 |
| **Service Worker 保活 / 唤醒系统 / 自启 Chrome** | Chrome 没开就不跑，文档明示 |
| **审计日志后端** | 本地 `chrome.storage.local` 的运行历史就够 |
| **URL 黑名单** | `safety.ts` 关键字守护已经够；多一层是防御性代码 |
| **微信支付 / 支付宝直连** | 要营业执照 + 对公账户，跟产品节奏不匹配。永远走 Creem / Paddle 这类海外 MoR |
| **云端 MCP relay**（远程 agent 连本地浏览器） | 违反"Chrome 必须开着 + 用户已登录"模型；如果有人真要远程跑数据收集，那是另一个产品 |
| **团队空间 / 共享脚本库 / 多用户协作** | 本产品是「个人 AI agent 的数据抓取工具」。要团队是另一个产品 |

## 付费墙

| | 免费版 | 付费版（¥30/月 或 ¥288/年） |
|---|---|---|
| 录制 / 重放 / 定时 / 本地下载 | ✅ 无限 | ✅ 无限 |
| 脚本数量 | 无限 | 无限 |
| MCP 接入（让 AI 调用） | ❌ | ✅ |
| 多设备脚本云同步 | ❌ | ✅ |

- **新用户 14 天 MCP 全功能试用**——让用户先尝到 AI 自动化的甜头
- 试用结束 / license 失效 → 扩展继续完整可用，仅 MCP 接口拒绝调用并返回明确升级提示
- 后端不可达 / 校验暂时失败 → 扩展给 **7 天宽限期**，避免后端事故牵连用户
- ❌ 不做按调用次数计费——quota 阈值都是拍脑袋；定位「水电免费 / AI 接入付费」更清晰

## 为什么扩展里不调 LLM

短结论：**录制 → 重放主链路里加 LLM 是负价值**。AI 应该在调用方，不在被调方。

| 维度 | 在扩展里加 LLM 的代价 |
|---|---|
| 隐私 | 用户后台数据（销售额、粉丝、对账）传给 LLM。电商 / 自媒体用户的命根子 |
| 成本 | 扩展不能安全嵌 API key，必须后端代理。账单可怕 |
| 延迟 | 每动作多 2-5 秒 LLM 等待 |
| 不确定性 | 同 DOM 同 prompt，今天点 A 明天点 B |
| 风险 | 猜错可能点到「删除」「支付」。不可逆动作代价远大于让用户重录 |

所有这些代价，**让用户自己的 AI agent 来承担反而合理**——agent 已经为自己的延迟 / 隐私 / 成本买单了。

## 已接受的风险（V1 不预先解决）

| 风险 | 处理 |
|---|---|
| Chrome 没开 → 定时任务不跑 | popup 里提示用户保持 Chrome 开启 |
| 平台改版让脚本失效 | 报错让用户重录，不自动恢复 |
| 平台未来检测自动化 DOM 操作 | 出现了再具体应对，**不预先做混淆 / 反检测** |
| 用户本地脚本含敏感选择器 | 不出本地就不需要加密；上云同步时再加 |
| `chrome.storage.local` 5MB 上限 | 几百条脚本足够；超出再迁 IndexedDB |
| 文件归档只能落在 `Downloads/` 子目录下 | Chrome 扩展 API 限制；用户想去 Desktop 自己手动移 |
| 后端挂了 → license 校验失败 | 扩展给 7 天宽限期 |
| 国内 / 海外用户付款渠道差异 | MVP 阶段手动收款 + 发码，规模化再上 Creem / Paddle |

## 何时更新本文档

- 增删功能 → 更新「P1」/「P2」/「永远不做」
- 商业模式 / 付费墙变动 → 更新「付费墙」
- 接受新风险 / 解决了已知风险 → 更新「已接受的风险」
- 新决策需要落档 → 写到 `docs/decisions.md`

> **重要原则**：「永远不做」表里的项目就是**永远不做**。如果讨论中又冒出来这些功能（"加个失败截图吧"、"飞书通知挺方便"、"步骤编辑器很多 RPA 都有"），先回这张表。要推翻某一项需要明确的产品定位变更，不是"试一下"。
