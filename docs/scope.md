# 当前范围（V1）

## V1 做什么

1. **录制**：用户在 popup 里点「开始录制」→ 选择当前 tab → 操作页面 → 点「停止」→ 命名保存
2. **重放**：手动触发或 `chrome.alarms` 定时触发，按选择器找元素并执行
3. **下载归档**：监听 `chrome.downloads`，按脚本配置归类到 `Downloads/webxport/{脚本名}/{日期}/` 下
4. **运行历史**：成功 / 失败 / 失败原因，存 `chrome.storage.local`

## V1 明确不做

> 这些不是"还没做"，是**有意识地不做**。LLM 写代码时不要擅自添加。

| 不做 | 原因 |
|---|---|
| LLM / AI 任何调用 | 隐私、成本、不确定性，详见下文 |
| 后端服务器、用户账号、云同步 | V1 验证产品形态，不引入运维 |
| 共享脚本网络 / 脚本市场 | 每个用户录自己的就够了；网络效应不是 V1 命题 |
| 自动登录 / 验证码处理 | 前提是用户已登录 |
| 数据上传到第三方（飞书/Notion/Slack） | 用户自己接，不与下游集成竞争 |
| Service Worker 保活 / 唤醒系统 | Chrome 没开就不跑，告知用户即可 |
| 数据分析 / 可视化 / 报表生成 | 用户拿到 xlsx 自己喂给 AI |
| 付费墙 / 激活码 / 试用期 | V1 免费 |

## 为什么 V1 没有 LLM

短结论：**录制 → 重放主链路加 AI 是负价值**。

| 维度 | 加 LLM 的代价 |
|---|---|
| 隐私 | 用户后台数据（销售额、粉丝、对账）传出浏览器去给 LLM。电商/自媒体用户的命根子，知道了会跑路 |
| 成本 | Chrome 插件不能安全嵌 API key，必须有后端代理。后端 = 一整套服务要部署/扩容/防滥用。每日定时 × N 用户 × M 网站，账单可怕 |
| 延迟 | 每动作多 2-5 秒 LLM 等待，10 步操作多 30 秒 |
| 不确定性 | 同 DOM 同 prompt，今天点 A 明天点 B |
| 风险 | 猜错可能点到"删除"或"支付"。这种不可逆动作的代价远大于让用户重录 |

LLM 在产品里**未来**可能有的位置（V2 之后）：
- **降级兜底**：选择器全失败时，可选地多调一次 LLM 推荐新选择器（在已经决定要报错的边界上多试一次）
- **云端脚本合并**：等同站点累积 N 份脚本后，batch 任务里合成"权威版本"——不在用户路径上

## 已接受的风险（V1 不预先解决）

| 风险 | 处理 |
|---|---|
| Chrome 没开 → 定时任务不跑 | popup 里提示用户保持 Chrome 开启 |
| 平台改版让脚本失效 | 报错让用户重录，不自动恢复 |
| 平台未来检测自动化 DOM 操作 | 出现了再具体应对，**不预先做混淆 / 反检测** |
| 用户本地脚本含敏感选择器 | 不出本地就不需要加密 |
| `chrome.storage.local` 5MB 上限 | 几百条脚本足够；超出再迁 IndexedDB |
| 文件归档只能落在 Downloads/ 子目录下 | Chrome 扩展 API 限制；用户想去 Desktop 就自己手动移 |

## 未来路线（V2+，不在 V1 实现）

按可能性递减排：

1. **跨设备脚本同步**：用户登录 → 脚本上云。前提是先验证 V1 留存
2. **运行报告推送**：跑完一次任务后推飞书/钉钉/邮件。属于"下游集成"，谨慎引入
3. **付费机制**：方案待定，候选：Chrome Web Store 付费 / 外部激活码 / 订阅制（依赖账号系统）
4. **共享脚本网络**：当且仅当用户量到一定规模才有意义。**目前不为这件事提前做架构准备**
5. **LLM 兜底**：选择器全失败时可选调 LLM。需要后端代理

## 何时更新本文档

- 增删 V1 功能 → 更新「V1 做什么」/「V1 明确不做」
- 接受新风险 / 解决了已知风险 → 更新「已接受的风险」
- 新决策需要落档 → 写到 `docs/decisions.md`
